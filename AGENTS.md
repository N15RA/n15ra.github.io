# 使用指南

## 原則

- 為了有更好的程式碼處理能力，請記得使用 context7 mcp 作為 Coding / Review 的 Basic，或者說在所有的操作之前，都應該使用 context7 做確認。
- 請遵循相關 Best Practice，且這個 Best Practice 是目前最新的。
- 不違反 KISS 原則（Keep It Short and Simple）。
- 不違反 DRY 原則（Don't Repeat Yourself）。
- 保持程式碼的「低耦合度」與「高內聚性」。
- 請確保專案永遠保持對於後續的協作維護者來說友善的環境，整個專案要夠清楚、好了解，但沒有過度的註解。
- 請確保你的檢視有全面仔細的整個專案的每一個專案結構與腳本、程式碼。
- 請不要隨意產生不必要的中間文件，總結也請乾淨明確，不用額外撰寫總結文件。
- 在回答時如果資料不足或你不確定，請直接標明為「不確定」或「需要查證」，不要補齊或編造細節，如果有需要跟我確認來進一步離解與梳理需求的話，可以隨時跟我確認。
- 希望你在準備好結論後，能再用另一個角度自我檢查一次，如發現可能錯誤或矛盾，可以修正並說明你為何改變原本想法。
- 不接受任何臨時性的調整與方案。
- 所有調整請一定需要先跟使用者確認，並且在確認後再進行調整。
- 為了確保程式碼、腳本、環境的穩定性，能不要 icon / emoji 就不要使用，除非是設計師所設計的例外。
- 如果必要的輔助工具沒有辦法使用，你可以在詢問過使用者之後安裝，不需要一直重新嘗試使用新的工具。
- 請確保所有的操作都符合資安規範，並且不會有任何資安風險。
- 請確保每一次的修正都不會造成原本正常的功能的損毀。
- 請注意，在產生所有文件、訊息時，請確保不要產生對文件無幫助的 Generated with / Co-Authored-By / Powered by 等等無幫助的資訊。
- 為了更好的追蹤與管理，除非任務非常簡單，否則請務必遵守創建新的 branch 來進行所有的修改，並且在完成後進行 Pull Request，來優化整個專案的運作流程，並提供一個夥伴 Code Review 的機會。
- 再完成所有變動、調整後，請確認是否有 formatter 可以進行程式碼格式化，並且執行一次，確保程式碼風格的一致性。
- 在提交 PR 之前，請務必確保所有測試通過。
